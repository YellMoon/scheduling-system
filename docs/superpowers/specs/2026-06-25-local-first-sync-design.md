# 格物工坊本地优先多端数据架构设计

日期：2026-06-25  
状态：已确认总体方向，待书面审阅后进入实施计划  
关联总任务：[task.md](../../../task.md)

## 1. 背景与目标

格物工坊要从单机桌面应用升级为多端系统，但最高优先级不是把所有数据搬到云端，而是降低服务器成本、保证桌面端断网可用，并让题库大文件由本地移动硬盘承载。

目标状态：

- 一台指定电脑作为“本地数据主机”，保存全量权威业务数据。
- 题库数据和题库附件独立存储在连接到主机的移动硬盘中。
- 其他电脑可安装完整桌面端，拥有全部桌面权限，断网时本地可写，联网后经用户确认再同步。
- 阿里云部署服务端，但只承担认证、设备、心跳、中继、快照、小程序 API 和任务队列，不作为最高权威数据库。
- 微信小程序大部分只读；仅允许财务个人资产导入、题库选题组卷、Word/PDF 导出等有限写入任务。
- 底层架构完成后，再逐步确认小程序 UI，最后部署阿里云和小程序。

## 2. 当前项目基础

代码库已经有可复用地基：

- 桌面端为 React + Electron。`public/electron.js` 启动内嵌 `backend`，当前通过 `DB_PATH` 指定 SQLite 路径，通过 `QUESTION_BANK_UPLOAD_DIR` 指定题库上传目录。
- 后端为 Node.js + Express + `better-sqlite3`。`backend/src/routes/sync.js` 已有 `/api/sync/pull`、`/api/sync/push`、`/api/sync/status`。
- `backend/src/schema.sql` 已有 `sync_log`、`sync_audit_log`、`outbox_events`，并且多张表已有 `tenant_id`、`updated_at`、`deleted` 等同步基础字段。
- 桌面端已有 `src/services/syncEngine.ts`，可以保存 pending changes、push、pull 和基础冲突结果。
- 小程序已有 `miniapp/src/utils/syncEngine.ts` 与 `miniapp/src/utils/sync.ts`，说明小程序侧同步框架已有雏形。
- 题库已包含导入、预览、编辑、组卷、Word 导出等功能；题库图片当前通过 `src/services/questionAssetStore.ts` 存入 IndexedDB。
- 代码库已有 `gateway/`、`miniapp/`、`modules/`，可支撑后续阿里云和小程序部署。

设计原则是“沿现有能力收敛”，不重写整个平台。

## 3. 方案比较

### 方案 A：阿里云作为主数据库

所有电脑和小程序都直接同步到阿里云，云端保存全量权威数据。

优点：

- 多端模型简单，云端天然可达。
- 小程序访问最直接。

缺点：

- 服务器、存储、备份和流量成本更高。
- 题库图片、Word、PDF 等大文件会持续推高云成本。
- 用户明确希望减少服务器成本，与目标不匹配。

结论：不采用。

### 方案 B：每台电脑完全对等同步

所有电脑都是平等节点，任意节点都可能成为数据权威。

优点：

- 理论上去中心化，任一电脑离线不影响其他电脑。

缺点：

- 冲突模型复杂，财务、课消、题目正文等高风险数据很难安全自动合并。
- 移动硬盘题库可能被多台电脑直接写入，容易损坏或产生附件孤儿。
- 对当前代码改造量过大。

结论：不采用。

### 方案 C：本地主机权威 + 云端中继/缓存

一台指定电脑保存权威数据；其他电脑离线写本地队列，联网后经用户确认同步到主机；主机再发布快照和任务状态到阿里云；小程序通过阿里云读快照和提交有限任务。

优点：

- 满足低服务器成本。
- 离线桌面端可用。
- 题库移动硬盘有单一写入入口。
- 高风险数据冲突可以集中在主机审核。
- 与当前 `backend`、`syncEngine`、`gateway`、`miniapp` 基础最匹配。

缺点：

- 主机需要尽量在线。
- 主机离线时小程序只能看到最近快照，写任务会排队。
- 局域网/公网连接、设备授权、冲突审核需要新实现。

结论：采用方案 C。

## 4. 系统角色

### 4.1 `primary-host`

指定本地数据主机。

职责：

- 保存权威 `scheduling.db`。
- 连接并管理移动硬盘题库目录。
- 接收其他电脑同步请求。
- 校验设备、同步授权和数据版本。
- 应用无冲突变更。
- 暂存冲突变更并展示给用户审核。
- 向阿里云发布只读快照、心跳和任务处理结果。

限制：

- 主机是唯一可以直接写移动硬盘题库目录的运行角色。
- 主机离线时，云端不得伪装成权威库。

### 4.2 `desktop-client`

其他电脑。

职责：

- 本地保存完整可用缓存。
- 断网时允许对教务、题库、财务等桌面端模块进行本地修改。
- 每次修改写入本地数据库和本地操作队列。
- 恢复网络后提示用户确认同步。
- 获得同步授权后推送变更到主机。
- 拉取主机已确认的最新变更。

限制：

- 不直接写主机移动硬盘。
- 不静默推送离线变更。
- 同步失败时必须保留 pending queue。

### 4.3 `cloud-relay`

阿里云服务端。

职责：

- 用户登录、小程序登录和权限判断。
- 设备注册。
- 主机心跳。
- 保存主机发布的只读快照。
- 保存小程序有限写入任务。
- 在主机离线时保存任务队列和数据快照时间。

限制：

- 不作为最高权威业务数据库。
- 不直接接受小程序对核心业务表的修改。
- 不长期保存题库原始大附件，除非是导出任务必要的临时文件。

### 4.4 `miniapp`

微信小程序。

职责：

- 读取教务、题库、财务统计等快照。
- 提交财务个人资产导入文件任务。
- 搜索/筛选题库快照。
- 选择试题、组卷、请求导出 Word/PDF。

限制：

- 不修改学生、课程、排课、付款、课消等核心教务数据。
- 不修改题目正文、答案、解析、知识点、模型、标签等题库基础数据。
- 不绕过云端权限直接访问主机。

## 5. 数据存储设计

### 5.1 桌面业务数据库

默认文件：

```text
<应用数据目录>/data/scheduling.db
```

主机可配置到其他路径。客户端本地也保存一份缓存数据库，但角色为 `desktop-client` 时，这份数据库不是最高权威库。

新增配置字段：

```json
{
  "nodeRole": "primary-host",
  "deviceId": "desktop_xxx",
  "hostBaseUrl": "http://192.168.1.10:3001",
  "cloudBaseUrl": "https://example.com",
  "mainDbPath": "D:/GewuData/scheduling.db",
  "questionBankPath": "E:/GewuQuestionBank",
  "questionAssetPath": "E:/GewuQuestionBank/assets"
}
```

配置保存位置：

```text
<Electron userData>/gewugongfang.config.json
```

环境变量映射：

- `mainDbPath` -> `DB_PATH`
- `questionBankPath` -> `QUESTION_BANK_ROOT`
- `questionAssetPath` -> `QUESTION_BANK_UPLOAD_DIR`
- `nodeRole` -> `GEWU_NODE_ROLE`
- `deviceId` -> `GEWU_DEVICE_ID`

### 5.2 题库移动硬盘目录

目录结构：

```text
<移动硬盘>/GewuQuestionBank/
  question-bank.db
  assets/
    images/
    word-imports/
    exports/
  backups/
  manifest.json
  lock.json
```

`manifest.json`：

```json
{
  "storeId": "qb_xxx",
  "schemaVersion": 1,
  "createdAt": "2026-06-25T00:00:00.000Z",
  "lastMountedByDeviceId": "desktop_xxx",
  "lastVerifiedAt": "2026-06-25T00:00:00.000Z"
}
```

要求：

- 题库路径必须通过 `manifest.json` 校验，不只依赖盘符。
- 移动硬盘未连接时，题库模块进入只读/不可用状态，并显示明确提示。
- 写操作必须检查 `lock.json` 和当前角色。
- IndexedDB 中的题库附件作为兼容缓存，不再作为长期权威存储。

## 6. 同步设计

### 6.1 操作模型

所有本地写入都转换为统一操作：

```ts
type SyncOperation = {
  operationId: string;
  tenantId: string;
  deviceId: string;
  tableName: string;
  recordId: string;
  action: 'create' | 'update' | 'delete';
  baseVersion: string | null;
  newVersion: string;
  payload: Record<string, unknown>;
  riskLevel: 'low' | 'medium' | 'high';
  createdAt: string;
};
```

`baseVersion` 是客户端修改前看到的记录版本。主机用它判断是否发生并发修改。

### 6.2 写入流程

桌面端每次写入：

1. 写本地业务表。
2. 写本地操作队列。
3. 更新本地同步状态。
4. 如果在线，不自动推送，只提示用户有待同步变更。

主机写入也进入操作日志，但主机操作可以立即标记为已应用，并发布给客户端/云端。

### 6.3 客户端推送流程

1. 客户端检测主机或云端可达。
2. 若 pending operation 数量大于 0，显示同步提示。
3. 用户确认后，请求同步授权。
4. 客户端先拉取主机最新版本，进行本地冲突预检。
5. 客户端推送 pending operations。
6. 主机返回：
   - `applied`: 已应用操作；
   - `conflicts`: 冲突操作；
   - `rejected`: 权限或 schema 错误；
   - `serverVersion`: 主机版本。
7. 客户端只清除已确认 applied 的操作。

### 6.4 冲突策略

低风险数据可自动合并：

- 备注追加。
- 非关键标签追加。
- 最近访问时间。

高风险数据必须人工确认：

- 付款。
- 课消。
- 资产流水。
- 排课时间。
- 学生余额。
- 课程定价。
- 题目正文。
- 题目答案。
- 题目解析。
- 题库附件替换。

主机同步控制台要显示：

- 本地版本。
- 客户端版本。
- 冲突字段。
- 操作来源设备。
- 操作时间。
- 可选处理：主机优先、客户端优先、字段合并、拒绝。

## 7. 阿里云设计

云端保存以下数据：

- 用户和小程序登录信息。
- 设备注册信息。
- 主机心跳。
- 同步授权令牌。
- 最近只读快照。
- 小程序任务队列。
- 临时导出文件元数据。

云端 API 分层：

- `/api/cloud/host/heartbeat`：主机心跳。
- `/api/cloud/snapshots/publish`：主机发布快照。
- `/api/cloud/snapshots/read`：小程序读取快照。
- `/api/cloud/tasks`：小程序提交允许的任务。
- `/api/cloud/tasks/:id/result`：小程序读取任务结果。
- `/api/cloud/devices`：设备注册与授权。

当主机离线：

- 小程序继续读取最近快照。
- 小程序界面显示快照时间。
- 新任务进入 `pending_host`。
- 涉及实时权威数据的动作显示“等待主机上线处理”。

- 移动硬盘断开时题库写操作被阻止。
- 移动硬盘连接时题库附件可保存到文件系统。

- 主机对无冲突操作应用成功。

## 12. 非目标

本阶段不做：

- 把全部题库附件长期上传阿里云。
- 让小程序直接改核心业务表。
- 多主机同时写同一个移动硬盘题库。
- 完整重写现有 `backend`、`gateway` 或 `miniapp`。
- 在底层同步稳定前做小程序视觉改版。

## 13. 开放风险

- 主机离线时间过长会导致小程序快照滞后。UI 必须显示快照时间。
- 离线客户端积累大量高风险修改时，冲突审核会变重。同步中心必须提供批量查看和筛选。
- 题库附件迁移需要兼容 IndexedDB 历史数据。实施时先保留兼容读取，再迁移写入路径。
- 当前部分排课数据仍使用 `localStorage`，实施统一变更捕获时要纳入迁移范围。
